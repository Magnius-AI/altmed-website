AWS ECS Fargate + RDS Deployment Guide
Project: Altmed website monorepo
Date checked against AWS docs: April 21, 2026

Detected stack in this repo:
- Frontend: Next.js 14 (`frontend/`)
- Backend: NestJS 10 (`backend/`)
- Database: PostgreSQL
- Images available to build now:
  - `frontend/Dockerfile`
  - `backend/Dockerfile`

Recommended AWS architecture for this repo:
- 1 Application Load Balancer (ALB)
- 2 ECS Fargate services
  - frontend service
  - backend service
- 1 Amazon RDS PostgreSQL instance
- 1 Amazon S3 bucket for uploaded images
- 2 Amazon ECR repositories
  - frontend image
  - backend image
- 1 ACM certificate for your subdomain
- 1 Route 53 alias record for the subdomain

Beginner-friendly note:
- This guide uses Fargate tasks in public subnets with tightly scoped security groups.
- That avoids NAT Gateway cost and keeps the setup simpler.
- RDS will still stay private in database subnets.
- After your first successful launch, you can move ECS tasks to private subnets later.

==================================================
ARCHITECTURE
==================================================

Traffic flow:
1. Browser opens `app.yourdomain.com`
2. Route 53 sends traffic to the ALB
3. ALB HTTPS listener terminates TLS with ACM
4. ALB routes:
   - `/api/*` -> backend target group
   - `/uploads/*` -> backend target group if you use local uploads fallback
   - everything else -> frontend target group
5. Backend connects to RDS PostgreSQL on port `5432`

Important repo-specific notes:
1. This repo works best with one public domain for both frontend and backend.
   Use:
   - `NEXT_PUBLIC_API_URL=https://app.yourdomain.com`
   - `API_URL_INTERNAL=https://app.yourdomain.com`
   - optional `BACKEND_URL_INTERNAL=<backend service URL>` for frontend server-side upload proxy calls
   - `FRONTEND_URL=https://app.yourdomain.com`

Future-safe domain strategy:
- Start with `app.altmedfirst.com` now if that is easier.
- Later, if you want the main site on `altmedfirst.com`, keep the same ECS, ALB, RDS, and S3 setup.
- Only the certificate, DNS target, and app URL env values need to change.
- Treat all public URL settings as one value:
  - `PUBLIC_APP_URL`
  - today: `app.altmedfirst.com`
  - later: `altmedfirst.com`

2. Uploaded admin images should use S3 in production.
   This repo now supports S3-backed uploads when the S3 env vars are set.
   If those env vars are missing, it falls back to local `/uploads` storage for local development.

3. I added a production seed command for ECS:
   - `npm run seed:prod`

==================================================
STEP 1 - Create or verify your VPC
==================================================

Use the default VPC if you are brand new to AWS and want the fastest path.

You need:
- at least 2 public subnets in different Availability Zones for the ALB
- at least 2 subnets for RDS DB subnet group

Simplest path:
1. Go to `VPC`
2. Verify the default VPC exists
3. Note the VPC ID
4. Note at least 2 public subnets in different AZs

For RDS:
- you can use private DB subnets if already available
- if you only have the default VPC, AWS can still help you create the DB subnet group from available subnets

==================================================
STEP 2 - Create security groups
==================================================

Create these 4 security groups:

1. `altmed-alb-sg`
   Inbound:
   - 80 from `0.0.0.0/0`
   - 443 from `0.0.0.0/0`
   Outbound:
   - all

2. `altmed-frontend-sg`
   Inbound:
   - 3000 from `altmed-alb-sg`
   Outbound:
   - all

3. `altmed-backend-sg`
   Inbound:
   - 3001 from `altmed-alb-sg`
   Outbound:
   - all

4. `altmed-rds-sg`
   Inbound:
   - 5432 from `altmed-backend-sg`
   Outbound:
   - all

Do not open port `5432` to the public internet.

==================================================
STEP 3 - Create the RDS PostgreSQL database
==================================================

In AWS Console:
1. Open `RDS`
2. Click `Create database`
3. Choose `Standard create`
4. Engine: `PostgreSQL`
5. Template:
   - use `Free tier` if available
   - otherwise choose the lowest-cost dev/test option you are comfortable with
6. DB instance identifier: `altmed-postgres`
7. Master username: `altmed`
8. Set a strong password
9. Instance class:
   - start with something small like `db.t3.micro` if available for your account/region
10. Storage:
   - start around `20 GB`
11. Connectivity:
   - choose your VPC
   - create or choose a DB subnet group
   - Public access: `No`
   - VPC security group: `altmed-rds-sg`
12. Additional configuration:
   - Initial database name: `altmed`
13. Create database

Wait until status becomes `Available`.

Write down:
- DB endpoint
- DB port

Example:
- endpoint: `altmed-postgres.abc123xyz.us-east-1.rds.amazonaws.com`
- port: `5432`

==================================================
STEP 4 - Create the S3 uploads bucket
==================================================

Create a bucket for uploaded admin images.

Suggested name:
- `altmed-uploads-prod`

Recommended setup:
1. Create the bucket in the same region as ECS
2. Use a folder prefix like `uploads/`
3. Decide how browsers will read the images:
   - simplest: a dedicated bucket or prefix that is public-read
   - better long term: CloudFront in front of the bucket

If you use direct S3 image URLs with this repo, set:
- `AWS_S3_PUBLIC_BASE_URL`
- example: `https://altmed-uploads-prod.s3.us-east-1.amazonaws.com`

The backend role that talks to S3 needs:
- `s3:PutObject`
- `s3:DeleteObject`

Important ECS note:
- Put S3 permissions on the backend task role, not only the execution role.
- `executionRoleArn` is for ECS image pulls, logs, and task startup secrets.
- `taskRoleArn` is what the AWS SDK inside the backend container uses for S3 uploads.
- If the backend logs `Could not load credentials from any providers`, the backend task is missing a usable `taskRoleArn` or that role is missing S3 permissions.

==================================================
STEP 11 - Create IAM roles for ECS
==================================================

You need:
1. ECS task execution role
   Use the managed policy that lets ECS pull from ECR and write logs.
   In many accounts AWS creates `ecsTaskExecutionRole` for you.
   If you use Secrets Manager values in the task definition, this role also needs permission to read those secrets.

2. Optional task role
   If you use S3 uploads, create or select a task role for the backend app code.

For a first deployment:
- execution role is required
- task role is required when the backend calls S3 from app code

If you use the sample backend task definition in this repo:
- the execution role must be allowed to call `secretsmanager:GetSecretValue`
- if your secrets use a customer-managed KMS key, also allow `kms:Decrypt`
- for S3-backed uploads, the backend `taskRoleArn` needs `s3:PutObject` and `s3:DeleteObject` on the uploads bucket/prefix

==================================================
STEP 12 - Store app secrets
==================================================

Best practice:
- store sensitive values in AWS Secrets Manager

Create secrets for:
- database password
- JWT secret
- JWT refresh secret
- email password
- recaptcha secret

You can use Systems Manager Parameter Store too, but Secrets Manager is easier to reason about for app secrets.

For your first launch, these are the core values you need:
- `NODE_ENV=production`
- `PORT=3001` for backend
- `DATABASE_HOST=<your-rds-endpoint>`
- `DATABASE_PORT=5432`
- `DATABASE_NAME=altmed`
- `DATABASE_USER=altmed`
- `DATABASE_PASSWORD=<from secret>`
- `JWT_SECRET=<from secret>`
- `JWT_REFRESH_SECRET=<from secret>`
- `JWT_EXPIRES_IN=15m`
- `JWT_REFRESH_EXPIRES_IN=30d`
- `FRONTEND_URL=https://app.altmedfirst.com`
- `RECAPTCHA_SECRET_KEY=<from secret>`
- `EMAIL_HOST=smtp.gmail.com`
- `EMAIL_PORT=587`
- `EMAIL_USER=info@altmedfirst.com`
- `EMAIL_PASS=<from secret>`
- `CONTACT_FORM_RECIPIENT=info@altmedfirst.com`
- `ADMIN_ALERT_EMAIL=admin@altmedfirst.com`

Frontend env vars:
- `NODE_ENV=production`
- `BACKEND_URL_INTERNAL=http://<backend-service-or-internal-alb>:3001`
- `API_URL_INTERNAL=https://app.altmedfirst.com`
- `NEXT_PUBLIC_API_URL=https://app.altmedfirst.com`

Backend storage env vars:
- `AWS_REGION=us-east-1`
- `AWS_S3_BUCKET=altmed-uploads-prod`
- `AWS_S3_UPLOAD_PREFIX=uploads`
- `AWS_S3_PUBLIC_BASE_URL=https://altmed-uploads-prod.s3.us-east-1.amazonaws.com`

Do not set the S3 env vars until the backend ECS task has a task role with S3 access.
If S3 env vars are set but the task role is missing, image uploads fail with:
`Image upload failed: Could not load credentials from any providers`.

Domain env strategy:
- deploy now with:
  - `PUBLIC_APP_URL=app.altmedfirst.com`
  - `FRONTEND_URL=https://app.altmedfirst.com`
  - `BACKEND_URL_INTERNAL=http://<backend-service-or-internal-alb>:3001`
  - `API_URL_INTERNAL=https://app.altmedfirst.com`
  - `NEXT_PUBLIC_API_URL=https://app.altmedfirst.com`
- switch later with:
  - `PUBLIC_APP_URL=altmedfirst.com`
  - `FRONTEND_URL=https://altmedfirst.com`
  - `BACKEND_URL_INTERNAL=http://<backend-service-or-internal-alb>:3001`
  - `API_URL_INTERNAL=https://altmedfirst.com`
  - `NEXT_PUBLIC_API_URL=https://altmedfirst.com`

Why set both frontend API URLs to the public subdomain:
- this repo routes API calls through the same domain
- the ALB will forward `/api/*` to the backend
- this avoids needing internal service discovery on day one

STEP 15 - Create the Application Load Balancer
==================================================

In AWS Console:
1. Open `EC2 > Load Balancers`
2. Create `Application Load Balancer`
3. Name: `altmed-alb`
4. Scheme: `Internet-facing`
5. IP type: `IPv4`
6. Select your VPC
7. Select at least 2 public subnets in different AZs
8. Security group: `altmed-alb-sg`

Do not finish listeners until you have:
- ACM certificate
- target groups

==================================================
STEP 16 - Request ACM certificate for your subdomain
==================================================

In AWS Console:
1. Open `Certificate Manager`
2. Request a public certificate
3. Domain name:
   - `app.altmedfirst.com`
4. Validation method:
   - `DNS validation`
5. If your DNS is in Route 53, let ACM create the validation record
6. Wait for certificate status `Issued`

==================================================
STEP 17 - Create target groups
==================================================

Create 2 target groups:

1. Frontend target group
   - Type: `IP`
   - Protocol: `HTTP`
   - Port: `3000`
   - VPC: your VPC
   - Health check path: `/`
   - Name example: `altmed-frontend-tg`

2. Backend target group
   - Type: `IP`
   - Protocol: `HTTP`
   - Port: `3001`
   - VPC: your VPC
   - Health check path: `/api/health`
   - Name example: `altmed-backend-tg`

Use target type `IP` because ECS Fargate tasks use `awsvpc` networking.

==================================================
STEP 18 - Finish ALB listeners and rules
==================================================

Create listeners:

1. HTTP listener on port `80`
   Action:
   - redirect to HTTPS `443`

2. HTTPS listener on port `443`
   Certificate:
   - select the ACM certificate for `app.altmedfirst.com`
   Default action:
   - forward to frontend target group

Add HTTPS listener rules:
1. Rule for `/api`
   Forward to backend target group

2. Rule for `/api/*`
   Forward to backend target group

Default:
- frontend target group

==================================================
STEP 19 - Create the frontend ECS service
==================================================

In ECS Console:
1. Open cluster `altmed-cluster`
2. Create service
3. Launch type: `Fargate`
4. Task definition: `altmed-frontend`
5. Service name: `altmed-frontend`
6. Desired tasks: `1`
7. Networking:
   - choose the same VPC
   - choose public subnets
   - Security group: `altmed-frontend-sg`
   - Assign public IP: `Enabled`
8. Load balancing:
   - attach to the frontend target group
   - container: frontend
   - port: `3000`
9. Create service

==================================================
STEP 20 - Create the backend ECS service
==================================================

In ECS Console:
1. Create another service in the same cluster
2. Launch type: `Fargate`
3. Task definition: `altmed-backend`
4. Service name: `altmed-backend`
5. Desired tasks: `1`
6. Networking:
   - choose the same VPC
   - choose public subnets
   - Security group: `altmed-backend-sg`
   - Assign public IP: `Enabled`
7. Load balancing:
   - attach to backend target group
   - container: backend
   - port: `3001`
8. Create service

Why public IP enabled here:
- it avoids NAT gateway cost for your first deployment
- inbound traffic is still controlled by security groups
- backend is not directly exposed because only the ALB security group can reach port `3001`

STEP 21 - Seed the RDS database
==================================================

This repo now supports a production seed command:
- `npm run seed:prod`

Run it once as an ECS task:
1. In ECS, choose the backend task definition
2. Click `Run task`
3. Use Fargate
4. Same VPC, subnets, and backend security group
5. Override the backend container command to:
   `npm run seed:prod`
6. Run task
7. Watch logs in CloudWatch

If successful, the seed creates:
- admin user
- service pages
- FAQs
- blog seed data
- site settings

Default seeded admin:
- email: `admin@altmedfirst.com`
- password: `CHANGE_ME_ON_FIRST_LOGIN`

Change that immediately after deployment.

==================================================
STEP 22 - Point the subdomain to the ALB
==================================================

If DNS is in Route 53:
1. Open your hosted zone
2. Create a new record
3. Record name:
   - `app`
4. Record type:
   - `A`
5. Alias:
   - `Yes`
6. Route traffic to:
   - your ALB

If DNS is outside Route 53:
- create a `CNAME` for the subdomain pointing to the ALB DNS name

Example:
- `app.altmedfirst.com` -> `altmed-alb-123456.us-east-1.elb.amazonaws.com`

==================================================
STEP 26 - Recommended next improvements
==================================================

After the first successful deployment, do these next:

1. Move ECS tasks to private subnets
   Then add:
   - NAT gateway
   or
   - VPC endpoints for ECR, CloudWatch Logs, and related services

2. Put secrets fully in Secrets Manager
   Avoid plain environment strings in task definitions.

3. Put CloudFront in front of the uploads bucket
   This gives cleaner public image URLs and better caching.

4. Set up a CI/CD pipeline
   Example:
   - GitHub Actions builds images
   - pushes to ECR
   - updates ECS services

5. Enable RDS backups and tune retention

==================================================
PROJECT-SPECIFIC RISKS TO KNOW
==================================================

1. If you do not set the S3 env vars, uploads fall back to local container storage
   That is fine for local development but not reliable for Fargate production.

2. Seed task is separate from app task
   This is normal in ECS.
   Run the seed task once after the backend image is deployed and the DB is reachable.

3. Cookie security
   I updated the frontend auth cookie handling so cookies become `secure` in production.

==================================================
LATER ROOT-DOMAIN CUTOVER
==================================================

When you are ready to move from `app.altmedfirst.com` to `altmedfirst.com`:

1. Request or expand the ACM certificate to cover:
   - `altmedfirst.com`
   - optional `www.altmedfirst.com`

2. Update the ALB HTTPS listener to use that certificate.

3. In Route 53, create an alias `A` record for the root domain pointing to the ALB.

4. Update ECS env values:
   - `FRONTEND_URL=https://altmedfirst.com`
   - `API_URL_INTERNAL=https://altmedfirst.com`
   - `NEXT_PUBLIC_API_URL=https://altmedfirst.com`

5. Register new frontend and backend task definition revisions and deploy them.

6. Optional cleanup:
   - keep `app.altmedfirst.com` and redirect it to `altmedfirst.com`
   - or retire the subdomain after verification
